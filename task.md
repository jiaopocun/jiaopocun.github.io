# 村级在线博物馆（照片为主）GitHub Pages 落地方案（A 方案）

## 0. 方案结论

* 采用 **A 方案**：照片与网站同仓库，直接用 **GitHub Pages** 发布。
* 你的规模（约 200 张、每张约 3MB）理论上可行，但务必关注 GitHub Pages 的发布与带宽限制：**发布站点 ≤ 1GB**、部署超时、带宽软上限等。 ([GitHub Docs][1])
* 因为访问量不大，本方案优先“简单可维护”，不引入服务器、不依赖复杂构建。

---

## 1. 目标与体验标准

### 目标

* 用“**系列标题（专题/相册）→ 照片小标题（单张展品）**”的结构，把照片做成“展览”而不是“散相册”。

### 体验标准（MVP）

* 首页能看到所有“系列卡片”（系列标题 + 一句话副标题 + 时间范围 + 地点 + 数量）。
* 点击系列进入照片墙（每张图下方显示统一格式的小标题）。
* 点击单张照片进入详情页（大图 + 信息卡：时间/地点/看点/补充说明）。
* 手机端可用、加载不至于崩。

---

## 2. 信息架构（页面与路由）

### 页面清单（最小集）

1. **首页（系列列表）**
2. **系列页（照片墙）**：按系列展示所有照片
3. **照片页（详情页）**：单张大图 + 信息卡

> 可选（后续迭代再做）：标签页、时间线页、地图页、投稿页。

---

## 3. 内容规范（参考你说的“1、2”）

### 3.1 系列标题（大标题）规范

系列标题默认来自文件夹名；以下字段可选，暂时不做可留空：

* 系列标题（8–16 字）
* 一句话副标题（20–40 字，说明这一组照片讲什么）
* 时间范围（例如“约1980–1995”）
* 地点（例如“老街一带”）
* 照片数量（自动统计也行）
* 封面图（系列里最“标志性”的一张）

### 3.2 照片小标题（统一格式）

缩略图下方的小标题统一为三段式：

* **时间｜地点｜看点一句话**
  缺信息就用“约/待考/待补充”占位，保持整站一致性。

示例：

* 约1986｜老街口｜村里第一次有电视机的合影
* 1992｜小学操场｜那年运动会接力赛冠军班

---

## 4. 仓库结构（建议）

> 核心原则：**路径稳定、命名一致、相对路径可用**（避免 GitHub Pages 子路径问题）。

推荐目录：

* 根目录：站点文件（HTML/CSS/少量 JS）
* `data/`：照片 CSV（只有一种格式，不再维护 series 表）
* `photos/<series-slug>/`：该系列的照片文件（文件夹名即系列标识）

命名建议：

* `series-slug`：文件夹名即系列名（建议英文/拼音+短横线，避免空格与特殊字符）
* 照片文件名：`1.jpg`、`2.jpg`、`3.jpg`…（统一扩展名，避免中文空格）
* 只上传文件夹时默认规则：**文件夹名 = 系列标识**，照片细节信息从 CSV 读取

---

## 5. 数据组织（仅照片 CSV / 数据库）

### 5.1 只维护照片 CSV（不再维护系列表）

所有照片信息（除系列归属外）只来自 CSV，系列归属只来自文件夹名。

**CSV 列顺序：**

* `photo_id`
* `time`
* `place`
* `highlight`
* `source`

示例行（对应图片 `1.jpg`）：

```csv
1,1980s,三组,农具,来自张三家
```

> 只需要一种 CSV 模板（例如 `1.csv`），不再有 series 表或 series 字段。

### 5.2 CSV 命名与绑定规则

* 系列归属：以 `photos/<series-slug>/` 文件夹名为准
* CSV 绑定：`data/<series-slug>.csv` 与 `photos/<series-slug>/` 一一对应
* 图片文件名：默认 `photo_id + 扩展名`（如 `1.jpg`）
* 照片小标题：`time | place | highlight`，缺字段用“约/待考/待补充”占位

### 5.3 可选：SQLite 小数据库

如果更偏“数据库式管理”，可用一个轻量 SQLite 文件（例如 `data/archive.sqlite`），表结构与上面一致。发布时导出为 CSV，供站点读取。

---

## 6. 更新流程（不写代码版）

1. 新建/确认系列文件夹：`photos/<series-slug>/`
2. 将照片放入系列文件夹，命名为 `1.jpg`、`2.jpg`…（统一扩展名）
3. 在 Excel/数据库中新增或更新该系列照片记录（只需照片表）
4. 导出为 CSV，放到 `data/`（文件名 = 系列文件夹名）
5. 提交并推送，等待 GitHub Pages 自动发布

---

## 7. 图片策略（A 方案下的现实建议）

你说每张约 3MB、访问不多——可以先不做复杂优化，但要知道这些硬约束：

* GitHub Pages **发布站点整体 ≤ 1GB**；带宽软上限等也要留意。 ([GitHub Docs][1])
* GitHub 普通仓库 **单文件 >100MiB 会被阻止**；网页端上传单文件 ≤25MiB（用 git 推送不受 25MiB 影响）。 ([GitHub Docs][2])

**建议（仍保持“简单”）：**

* 先用原图直接上线（3MB×200≈600MB），但给自己留缓冲：不要把站点堆到 900MB+。
* 如果后面感觉加载慢，再加“缩略图/大图两套”是最划算的优化（不用改结构，只改图片引用逻辑）。

## 8. GitHub Pages 发布流程（不需要 Actions）

### 7.1 发布方式

使用“从分支发布”：

* Source：Deploy from a branch
* Branch：`main`
* Folder：`/(root)` 或 `/docs`

GitHub 官方允许把发布源设为某个分支的根目录或 `/docs`。 ([GitHub Docs][3])

### 7.2 发布后的访问地址

* 项目站：`https://<username>.github.io/<repo>/`

> 方案里使用相对路径，能最大程度避免“子路径 base”问题。

## 9. 验收清单（上线前 10 分钟自检）

### 内容一致性

* [ ] 每个系列都有：标题 + 副标题 + 时间范围 + 地点 + 封面
* [ ] 每张照片都有：时间/地点/看点（三段式小标题不空）
* [ ] 表格/数据库里的 `series_slug` 与文件夹名完全一致
* [ ] `photo_id` 与实际文件名（1..n）匹配且无重复

### 路径与资源

* [ ] 首页 → 系列页 → 照片页链路全通
* [ ] 所有图片都能在 GitHub Pages 域名下直接打开
* [ ] 站点总大小明显 < 1GB（给后续留空间） ([GitHub Docs][1])

### 体验

* [ ] 手机打开系列页不会一次性卡死（至少有懒加载/分页/分批渲染其一）
* [ ] 图片加载失败时有占位或提示（避免“空白瀑布流”）

## 10. 后续迭代路线（按收益排序）

1. **搜索/筛选**：按地点/年代/人物标签筛
2. **时间线入口**：按年代浏览
3. **地图入口**：按地点点位浏览
4. **投稿/纠错**：一个简单表单（哪怕先用外部表单链接）
5. **两套图（thumb/web）**：提升加载速度与体验

---

## 11. 风险与注意事项

* 不要放敏感个人信息（身份证号、联系方式、未成年人信息等）。
* 人物照片/采访建议明确授权（“同意线上展示/转载规则”）。
* 不要把站点体积逼近 1GB 上限，否则后续迭代会很痛。 ([GitHub Docs][1])
