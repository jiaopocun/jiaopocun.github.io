# 村级在线博物馆（照片为主）具体实施方案（v1，无代码）

> 目标：在 GitHub Pages 上落地一个“系列 → 照片 → 详情”的在线照片博物馆，结构清晰、内容可持续维护，首版以最少技术栈完成上线。

---

## 1. 目标与范围（v1）

### 目标
- 以“系列（专题/相册）”为一级，单张照片为二级，形成“展览式”浏览体验。
- 站点可在手机端正常使用，加载不崩溃。
- 内容管理简单：维护系列文件夹与对应 CSV 即可更新。

### 范围（只做必需功能）
- 首页：系列卡片列表。
- 系列页：该系列的照片墙。
- 照片详情页：大图 + 信息卡（时间/地点/看点/补充说明）。

---

## 2. 信息架构与页面关系

- 首页：展示全部系列卡片（标题、副标题、时间范围、地点、数量、封面）。
- 系列页：按系列展示所有照片缩略图 + 三段式小标题。
- 照片页：单张大图 + 信息卡（时间/地点/看点/补充说明）。

---

## 3. 内容规范（统一口径）

### 3.1 系列信息（来自系列文件夹）
每个系列至少包含：
- 系列标题（文件夹名为系列标识）
- 一句话副标题（20–40 字）
- 时间范围（如“约1980–1995”）
- 地点（如“老街一带”）
- 照片数量（自动统计）
- 封面图（系列内一张代表性照片）

### 3.2 照片小标题（三段式）
统一格式：时间｜地点｜看点一句话  
缺项统一用“约/待考/待补充”占位，确保全站一致。

示例（文字说明，不是代码）：
- 约1986｜老街口｜村里第一次有电视机的合影
- 1992｜小学操场｜那年运动会接力赛冠军班

---

## 4. 数据组织（仅维护照片 CSV）

### 数据原则
- 不再维护系列表，系列归属由文件夹名决定。
- 每个系列只维护一份 CSV。

### CSV 字段（固定顺序）
photo_id、time、place、highlight、source

### 绑定规则
- 系列文件夹路径：photos/<series-slug>/
- 系列 CSV 路径：data/<series-slug>.csv
- 照片文件名：photo_id + 扩展名（如 1.jpg）

---

## 5. 目录结构（落地目录）

- 根目录：站点文件（HTML/CSS/少量 JS）
- data/：每个系列一份 CSV
- photos/<series-slug>/：系列照片文件夹

命名约定：
- series-slug：短横线命名，避免空格与特殊字符
- 照片：1.jpg、2.jpg、3.jpg…

---

## 6. 工作流程（无代码版）

1. 新建系列文件夹：photos/<series-slug>/
2. 将照片放入系列文件夹，并统一命名为 1.jpg、2.jpg…
3. 在 CSV 中填写该系列全部照片信息（仅照片表）
4. CSV 放入 data/，文件名与系列文件夹一致
5. 提交并推送，等待 GitHub Pages 自动发布

---

## 7. 体验与性能底线

- 手机端可滑动浏览，不出现长时间空白。
- 列表页与详情页均可返回。
- 图片加载失败时有占位提示或默认图。

---

## 8. GitHub Pages 发布方式

- 发布源：main 分支
- 发布目录：根目录或 /docs
- 访问地址：https://<username>.github.io/<repo>/

---

## 9. 里程碑与任务拆分（建议顺序）

### M1 内容准备
- 确定系列数量与名称
- 准备每个系列的封面与基础描述
- 照片重命名并归档

### M2 数据整理
- 完成每个系列 CSV
- 统一三段式小标题信息

### M3 站点结构
- 首页/系列页/详情页结构确认
- 导航与跳转逻辑确认

### M4 发布与验收
- GitHub Pages 配置
- 全链路点击检查
- 体积与性能自检

---

## 10. 验收清单（上线前）

- 每个系列都有标题、副标题、时间范围、地点、封面图
- 每张照片都有时间/地点/看点
- 系列文件夹名与 CSV 文件名一致
- photo_id 与照片文件名完全对应
- 首页 → 系列页 → 详情页链路可用
- 站点体积小于 1GB

---

## 11. 风险与注意事项

- GitHub Pages 发布上限与带宽软限制需要留余量。
- 大图加载慢时再考虑双版本图片（缩略图 + 大图）。
- 人像与敏感信息需授权与审慎处理。
